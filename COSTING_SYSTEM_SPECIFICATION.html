Customer Override Fields

These fields are optional. When present, they override the master-driven defaults in all calculations.

FieldBehaviour
Customer Inter-Ply Thickness Overrides fabric_rate_master.thickness_mm. Entered when customer specifies their own inter-ply rubber thickness requirement.
Customer Carcass Thickness Overrides fabric_strength_master carcass value. If present, this value is used in ALL calculations instead of the master value.
Customer Total Belt Thickness (TBT) A hard ceiling constraint. Calculated / actual belt thickness must NOT exceed this value. All components (carcass + covers + other layers) must fit within TBT.
Constraint Rules — LOCKED
  • Customer Inter-Ply Thickness → overrides fabric_rate_master default
  • Customer Carcass Thickness → overrides master carcass thickness
  • Customer TBT → hard ceiling; Σ all components ≤ TBT
  • When BOTH Customer Carcass Thickness AND Customer TBT are specified: both apply simultaneously; carcass must equal the customer carcass value AND total must stay within TBT. Remaining thickness budget for covers = TBT − customer_carcass_thk. If covers + carcass > TBT → flag as infeasible.

Key Distinction — Individual Ply Rating vs. Per Ply Rating

These are two different numbers — never confuse them
TermDescriptionExampleSource
Individual Ply Rating (Quotation) Commercial / rounded values used on customer-facing documents 100, 125, 160, 200… individual_ply_rating_master
Per Ply Rating (Actual) Exact engineering value = total_strength ÷ n_ply NN-315/3 → 315 ÷ 3 = 105.0 fabric_strength_master.per_ply_rating

Individual Ply Rating also serves as the LOOKUP KEY into fabric_rate_master to resolve GSM and inter-ply thickness for a given fabric type.

2.2 Four Carcass / Total Belt Thickness Scenarios

These four scenarios are mutually exclusive. Exactly one is active per costing line. The active scenario determines both carcass_thk and total_belt_thk.

1 No Customer Override (Master Values)
inter_ply_thk = As Per Fabric Rating Master carcass_thk = As Per Fabric Rating Master total_belt_thk = carcass_thk + top_cover_thk + bottom_cover_thk
2 Customer Specifies Inter-Ply Thickness
customer_inter_ply_thk = [entered by user] carcass_thk = master_carcass_thk + (n_ply − 1) × (customer_inter_ply_thk − master_inter_ply_thk) total_belt_thk = carcass_thk + top_cover_thk + bottom_cover_thk
Why (n_ply − 1)?
n plies of fabric create (n − 1) inter-ply rubber gaps. Each gap grows by the delta between customer and master inter-ply value. Carcass thickness grows by (n_ply − 1) × delta.
3 Customer Specifies Carcass Thickness Directly
carcass_thk = customer_carcass_thk (direct override) total_belt_thk = carcass_thk + top_cover_thk + bottom_cover_thk
4 Customer Specifies Total Belt Thickness (TBT)
total_belt_thk = customer_TBT (hard ceiling — cannot be exceeded) carcass_thk = customer_TBT − top_cover_thk − bottom_cover_thk
Constraint — LOCKED
If customer specifies BOTH carcass_thk (Scenario 3) AND TBT (Scenario 4) simultaneously, both constraints apply:
  • carcass must equal the customer carcass value
  • total must still stay within TBT
  • If carcass + top + bottom > TBT → flag as infeasible

Costing Engine — Calculation Rules

3.1 Two Formula Families LOCKED

Every layer in the belt belongs to exactly one of two formula families. Never mix the two formulas for the same layer.

Family 1 — Rubber Layers (SG-based)
layer_weight (kg) = W_eff (m) × thickness (mm) × SG (g/cm³) × L_eff (m)

Unit check: m × mm × g/cm³ × m → kg (conversion factor = 1)

SG comes from compound_master per compound code. Each rubber layer is an INDEPENDENT declaration. SG values from different layers are NEVER added or averaged.

Applies to: Top cover, Bottom cover, Fabric skim, Breaker skims, Sidewall, Cleat, Blinker / Edge

Family 2 — Textile Layers (GSM-based, NO SG)
layer_weight (kg) = W_eff (m) × (GSM ÷ 1000) × n_layers × L_eff (m)
Why no SG?
GSM (grams per square metre) already encodes mass-per-area directly. The fabric's thickness and density are baked into the GSM specification by the fabric supplier. Using SG on top would double-count the mass.

Applies to: Carcass fabric (n plies), Breaker fabric top, Breaker fabric bottom

3.2 Effective Width & Effective Length

Effective Width

W_eff = Belt Width + 3 cm (fixed wastage — always, regardless of fabric type)

The 3 cm is a flat addition, not a percentage.

Effective Length (wastage by fabric type)

Wastage is applied to Roll Length when calculating total material needed for an order. It accounts for fabric roll end trim, splicing loss, and handling.

Fabric TypeWastageL_eff Formula
NN+4%L_eff = Roll Length × 1.04
PP+4%L_eff = Roll Length × 1.04
EP+3%L_eff = Roll Length × 1.03
EE+3%L_eff = Roll Length × 1.03
SWTBC — values not yet confirmed
OthersTBC — values not yet confirmed

3.4 Cover Calculation — Two-Component Model LOCKED

Formula (per cover, calculated independently)

cover_weight (kg) = W_eff × thickness (mm) × SG × L_eff cover_cost (₹) = cover_weight × price_per_kg

Why Top and Bottom Are Always Independent

Different top and bottom compounds is a NORMAL customer requirement, not an edge case. Real-world scenarios:

1Cost optimisationPremium grade top (wear face), cheaper bottom
2FunctionalFire-resistant top, general-purpose bottom (mining)
3Heat resistanceHeat-resistant top, GP bottom (hot material)
4Asymmetric thicknessSame compound, different mm (most common case)

Display / PDF — Always Show Both Rows

Cover (Top) [compound] [mm] [kg] [₹/kg] [₹] Cover (Bottom) [compound] [mm] [kg] [₹/kg] [₹] ───── ───── Total Cover [kg] [₹]
NEVER collapse to a single "Cover" line — two rows always required
Reason: cost audit, quarterly rate comparison, customer queries.

What NOT to Do (will cause bugs)

  • ✗ Single "Cover" field with one compound picker
  • ✗ "Same as top" checkbox on bottom field
  • ✗ Average cover SG or average cover ₹/kg
  • ✗ One cover_compound_code column + asymmetric flag
  • ✗ One combined cover weight, then split for cost

fabric_wt (kg) = W_eff (m) × (GSM ÷ 1000) × n_ply × L_eff (m)
ValueSource
GSMfabric_rate_master.grades.gsm — Lookup key: fabric_type + individual_ply_rating
₹/kgfabric_rate_master.grades.price_per_kg (same lookup)

No SG involved. GSM already encodes mass-per-area. SG would double-count.

Same Applies to Breaker Fabric

breaker_fabric_wt = W_eff × (breaker_GSM ÷ 1000) × n_breaker_layers × L_eff GSM source: breaker_master

3.6 Skim Calculation — Remainder Method LOCKED

Decision: Remainder method preserved (legacy formula, fully locked)
  • Reason 1: Oracle validation numbers were produced this way.
  • Reason 2: Direct method needs skim_thickness_mm which does not exist in master data today.
  • Reason 3: A hybrid approach would introduce branching — violates the one universal engine rule.
Critical Naming Warning
The mind map labels one intermediate value "Total Belt Weight." This is NOT the actual total belt weight. They are two different numbers.
Mind Map TermEngine Variable NameWhat It Actually Is
"Total Belt Weight" belt_wt_without_breaker SANDWICH APPROXIMATION: belt modelled as single slab of cover compound
Final reported figure total_belt_weight Actual physical sum of all components

Using the wrong name in code will cause bugs. Always use belt_wt_without_breaker for the intermediate sandwich value.

The Four-Step Calculation

  1. Sandwich approximation (uses cover SG, NOT skim SG)
    belt_wt_without_breaker = W_eff × (carcass_thk + top_thk + bottom_thk) × cover_SG × L_eff

    The active carcass_thk (from whichever of the 4 scenarios applies) is used here.

  2. Independent component weights (already known)
    top_cover_wt = W_eff × top_thk × SG_top × L_eff bottom_cover_wt = W_eff × bottom_thk × SG_bottom × L_eff fabric_wt = W_eff × (GSM÷1000) × n_ply × L_eff
  3. Skim is the residual (SG_skim is NEVER used in weight)
    skim_wt = belt_wt_without_breaker − top_cover_wt − bottom_cover_wt − fabric_wt
    Important
    SG_skim does NOT appear anywhere in the skim weight calculation. SG_skim only appears in skim COST via compound_master.price_per_kg.
  4. Add breakers to get actual total belt weight
    total_belt_weight = belt_wt_without_breaker + breaker_fabric_wt (if BOT or BOB = Yes) + breaker_skim_wt (if BOT or BOB = Yes) + sidewall_wt (if sidewall belt) + cleat_wt (if chevron/cleated belt) + …
Open Question — Different SGs on Top and Bottom Covers
When top and bottom covers use different compounds (different SGs), the sandwich approximation in Step 1 uses a single cover_SG. It is not yet confirmed which value to use:
  • Top cover SG?
  • An average?
  • Top cover SG always (as the primary wear surface)?
*** MUST BE CONFIRMED BEFORE ENGINE IMPLEMENTATION ***

Locked Architectural Rules

The following rules have been explicitly confirmed and locked. They must not be changed without a deliberate design decision and re-documentation.

RULE 1 — SG Is Per-Layer, Never Combined

SG is a density (g/cm³) — a property of a specific material. It is physically impossible to add or average densities of different materials to get a meaningful combined density.

Two compounds = two SG lookups = two weight calculations = two kg values that are then added together.

  • ✗ combined_SG = SG_top + SG_bottom
  • ✗ avg_SG = (SG_top + SG_bottom) / 2
  • ✓ total_wt = top_cover_wt + bottom_cover_wt (kg + kg = kg)
RULE 2 — Top and Bottom Covers Are Always Two Independent Components

Even when 95% of belts use the same compound on both sides, the form always shows two separate fields, the engine always runs two separate calculations, and storage always holds two separate rows.

There is no "same as top" shortcut. There is no blending.

RULE 3 — Skim Weight Is Derived by Subtraction (Remainder Method)
skim_wt = belt_wt_without_breaker − fabric_wt − top_cover_wt − bottom_cover_wt

SG_skim does NOT appear in skim weight. It only appears in skim cost. The intermediate value belt_wt_without_breaker is NOT the same as total_belt_weight. They must never be named the same thing in code.

RULE 4 — One Universal Engine, No Per-Belt-Type Branching

There is no if (belt_type === 'chevron') { ... } anywhere in the engine.

belt_type_master declares the component set for each belt type. The engine loops over the declared components only.

Belt TypeAdditional components vs. base multiply belt
Multiply fabriccovers + carcass + fabric skim (+ breakers if used)
Chevronabove + cleats (+10% wastage on cleat)
Sidewallabove + sidewall (+10% wastage on sidewall) + cleats
Steep anglefull set
Bucket elevatorTBC
Straight warpcarcass handling TBC
Why this rule exists
The prior X5 architecture used per-type branching. This caused silent miscalculations when new belt types were added. The component declaration model in belt_type_master is the fix.
RULE 5 — Costing Grade Drives Calculations; Indus and Customer Grades Are Display-Only
GradeUsed For
Indus GradeIS standard designation → quotation document only
Customer GradeWhat the customer calls it → quotation document only
Costing GradeActual internal compound → SG, ₹/kg, all calculations

The engine only reads Costing Grade. It never reads Indus or Customer.

RULE 6 — GSM-Based Formula for All Textile Layers (No SG)

Fabric (carcass and breaker) uses GSM ÷ 1000 to get kg/m². SG is never applied to textile layers. GSM already encodes the mass. Applying SG to a textile layer would double-count the mass.

RULE 7 — Width Wastage Is Flat +3 cm; Length Wastage Is Type-Dependent %
  • Width: always add 3 cm flat (never a percentage)
  • Length: NN/PP → +4%;   EP/EE → +3%;   SW and others → TBC
RULE 8 — Snapshots at Quote Send

At the moment a quotation is sent, all compound codes, SG values, and ₹/kg rates are frozen as snapshot columns in the quotation line record. Future rate changes in master data do not affect sent quotes.

Minimum snapshot fields per compound layer:

compound_code sg_snapshot price_per_kg_snapshot weight_kg cost_inr
